# PlaylistAnalyze: подробный отчет по состоянию проекта

Дата отчета: 2026-05-23

Этот файл описывает, что происходило в проекте, что сейчас уже сделано, на чем построена система, как работает алгоритм рекомендаций, как запускать проект и что логично делать дальше.

Важно: в этом документе намеренно нет паролей, токенов и приватных учетных данных. Такие данные нельзя хранить в репозитории.

## 1. Суть проекта

PlaylistAnalyze - это веб-сервис для рекомендации треков, которые можно добавить в пользовательский плейлист.

Идея проекта:

1. Пользователь собирает или загружает список треков.
2. Backend пытается сопоставить эти треки с треками из словаря модели, обученной на Spotify Million Playlist Dataset.
3. Из найденных треков строится музыкальный контекст плейлиста.
4. Система выбирает похожие треки из MPD и ранжирует их.
5. Frontend показывает top-N рекомендаций с объяснениями, почему трек был предложен.

Проект не анализирует аудио, тексты песен, жанры напрямую и не обращается к Spotify API. Основной сигнал - совместная встречаемость треков в реальных плейлистах из MPD.

## 2. Датасет и модель

Основной датасет - Spotify Million Playlist Dataset, сокращенно MPD.

MPD содержит реальные плейлисты Spotify. Внутри каждого плейлиста есть треки с названиями, артистами, album name, Spotify URI и другой метаинформацией.

В проекте MPD используется так:

- один плейлист рассматривается как "предложение";
- один трек рассматривается как "слово";
- если два трека часто встречаются в похожих плейлистах, модель делает их embedding-векторы близкими;
- после обучения каждый трек получает числовой вектор;
- похожесть треков и плейлистов считается через скалярное произведение нормализованных векторов.

Текущая модель лежит в:

```text
data/models/item2vec_mpd_gensim
```

Для inference нужны основные файлы:

```text
item_vectors_normalized.npy
vocab.jsonl
model_meta.json
```

Что означает каждый файл:

- `item_vectors_normalized.npy` - матрица нормализованных Item2Vec-векторов треков.
- `vocab.jsonl` - словарь треков: URI, название, артист, альбом, count и индекс.
- `model_meta.json` - служебная информация о модели.

Сырые MPD JSON-файлы для работы сайта не нужны. Они нужны только для обучения или переобучения модели.

## 3. Алгоритм рекомендаций

### 3.1. Входные данные

На вход recommender получает список seed-треков. Это треки, которые пользователь уже добавил в плейлист и которые удалось найти в словаре MPD-модели.

Если пользователь загрузил TXT/CSV, сначала происходит matching:

```text
artist/title из файла -> поиск в vocab.jsonl -> track_uri модели
```

В рекомендации отправляются только уверенно найденные треки.

Если строка из файла не найдена в MPD, она показывается во frontend, но не используется в расчетах рекомендаций.

Если трек найден неуверенно, он помечается отдельно. Такие треки не должны тихо ломать рекомендации.

### 3.2. Векторы плейлиста

Из seed-треков строятся два основных вектора:

1. `whole_query`

Средний вектор всего распознанного плейлиста. Он отвечает за общий стиль плейлиста.

2. `recent_query`

Средний вектор верхних `recent_k` распознанных треков. В интерфейсе верх списка считается самым свежим концом плейлиста, поэтому именно эти треки отвечают за продолжение текущего направления.

Важно: `recent_k` считается по распознанным трекам, а не по всем строкам исходного файла. Если в загруженном плейлисте есть строки, которых нет в MPD, они не участвуют в recent window.

### 3.3. Candidate pool

Сначала система не ранжирует всю базу целиком финальной формулой. Она выбирает набор кандидатов - `candidate_pool`.

Раньше кандидаты искались через полный проход по всем векторам:

```text
vectors @ query
```

по кускам массива.

Сейчас добавлен быстрый путь через FAISS:

```text
FAISS IndexFlatIP -> top-k nearest tracks
```

Если FAISS установлен и индекс успешно собран, backend берет ближайшие треки через FAISS. Если FAISS недоступен, проект не падает и использует старый numpy-путь.

### 3.4. Фильтрация кандидатов

После получения кандидатов система исключает:

- seed-треки, которые уже есть в плейлисте;
- очень похожие дубликаты seed-треков, например другую версию того же трека того же артиста;
- треки ниже `min_count`, если режим не разрешает поднять их через favorite artist affinity.

`min_count` влияет только на рекомендации. Он не должен влиять на поиск трека по названию или артисту.

### 3.5. Финальная формула score

Для каждого кандидата считаются компоненты:

```text
recent_similarity
whole_playlist_similarity
multi_seed_support
popularity_prior
favorite_artist_affinity
```

Смысл компонентов:

- `recent_similarity` - насколько трек похож на последние распознанные треки плейлиста.
- `whole_playlist_similarity` - насколько трек похож на общий средний вектор всего плейлиста.
- `multi_seed_support` - насколько трек поддерживается сразу несколькими seed-треками, а не только одним.
- `popularity_prior` - мягкая поправка на популярность в MPD.
- `favorite_artist_affinity` - бонус трекам артистов, которые часто встречаются в seed-плейлисте.

Общая идея формулы:

```text
score =
  recent_weight * recent_similarity
+ whole_weight * whole_playlist_similarity
+ multi_seed_weight * multi_seed_support
+ popularity_weight * popularity_prior
+ favorite_artist_weight * favorite_artist_affinity
```

Весовые коэффициенты зависят от выбранного режима.

## 4. Режимы рекомендаций

Сейчас во frontend и backend есть 4 режима.

### 4.1. Balance

Сбалансированный режим.

Веса:

```text
recent:      45%
whole:       25%
multi-seed:  20%
popularity:  10%
favorite:     0%
```

Что делает:

- учитывает последние треки;
- учитывает весь плейлист;
- поднимает треки, которые похожи сразу на несколько seed-треков;
- немного учитывает популярность.

Это основной режим по умолчанию.

### 4.2. More recent

Режим продолжения последних треков.

Веса:

```text
recent:      65%
whole:       15%
multi-seed:  15%
popularity:   5%
favorite:     0%
```

Что делает:

- сильнее ориентируется на последние `recent_k` распознанных треков;
- меньше смотрит на весь плейлист;
- подходит, если пользователь хочет продолжить текущую концовку плейлиста.

### 4.3. Favorite

Режим любимых артистов.

Веса:

```text
recent:      25%
whole:       20%
multi-seed:  15%
popularity:   5%
favorite:    35%
```

Что делает:

- сильнее учитывает артистов, которые часто встречаются в seed-плейлисте;
- может поднимать треки этих артистов даже при небольшом `count`;
- дополнительно использует индекс артистов, чтобы быстро добавлять треки любимых артистов в candidate pool.

Этот режим полезен, если в плейлисте явно повторяются одни и те же исполнители и нужно найти еще треки этих исполнителей или близкий контекст.

### 4.4. More popular

Режим более популярных треков.

Веса:

```text
recent:      35%
whole:       20%
multi-seed:  20%
popularity:  25%
favorite:     0%
```

Что делает:

- сильнее поднимает треки, которые часто встречаются в MPD;
- подходит, если рекомендации должны быть более узнаваемыми и менее нишевыми.

## 5. Поиск треков

### 5.1. Как поиск работал раньше

Раньше поиск был простым:

1. Нормализовывался запрос.
2. Backend проходил по всему `vocab`.
3. Для каждого трека заново нормализовывались artist/title.
4. Проверялось substring-соответствие.
5. Результаты останавливались после `limit`.

Проблемы старого подхода:

- поиск мог быть быстрым на локальной машине, но тяжелым на слабом сервере;
- каждый запрос заново нормализовывал около 1.19 млн строк;
- `min_count` мог мешать поиску редких треков;
- не было нормальной пагинации "показать еще".

### 5.2. Как поиск работает сейчас

Сейчас при загрузке модели строится search index:

- нормализованные artist/title сохраняются в памяти один раз;
- строится token index: token -> список индексов треков;
- строится prefix lookup для быстрых префиксных запросов;
- добавлен result cache для повторных страниц выдачи.

Когда пользователь ищет трек:

1. Запрос нормализуется.
2. По токенам выбирается небольшой набор кандидатов.
3. Кандидаты ранжируются той же scoring-логикой.
4. Если запрос редкий и token index не нашел достаточно результатов, есть fallback на полный проход по уже нормализованным строкам.

`/tracks/search` принимает:

```text
q
limit
offset
```

`min_count` для поиска треков больше не используется.

### 5.3. Почему Juice WRLD мог не находиться

Проверка показала, что точного артиста `Juice WRLD` и некоторых ожидаемых треков нет в текущем `vocab.jsonl`.

Причина не в frontend и не в фильтре `min_count`, а в данных:

- MPD - старый датасет;
- если трека или артиста нет в словаре модели, backend не может вернуть точное совпадение;
- возможны только частичные совпадения по словам, если такие слова встречаются в базе.

## 6. Загрузка плейлиста и matching

Frontend умеет загружать TXT/CSV.

После загрузки:

1. Треки попадают в список на сайте.
2. Frontend отправляет их в `/recommend/match`.
3. Backend возвращает подробный результат matching для каждой строки.
4. UI помечает строки статусами:

```text
найдено в MPD
нет в MPD
неуверенное совпадение
не проверено
```

В рекомендации отправляются только найденные `track_uri`.

Это исправляет старую проблему, когда файл загружался в интерфейс, но рекомендации падали с ошибкой `422 Unprocessable Content` или frontend показывал `[object Object]`.

Ошибки API теперь форматируются человеком читаемо на frontend.

## 7. Backend

Backend сделан на FastAPI.

Основные файлы:

```text
backend/app/main.py
backend/app/config.py
backend/app/schemas.py
backend/app/api/health.py
backend/app/api/tracks.py
backend/app/api/recommendations.py
backend/app/services/model_store.py
backend/app/services/item2vec_service.py
backend/app/services/playlist_matcher.py
backend/app/services/recommender.py
```

### 7.1. `main.py`

Создает FastAPI-приложение, подключает:

- CORS;
- API routers;
- static frontend;
- lifecycle загрузки модели.

При старте создается `Item2VecService`, который загружает модель, search index, artist index и FAISS index.

Если модель не загрузилась, поведение зависит от `REQUIRE_MODEL_ON_STARTUP`.

### 7.2. `config.py`

Читает настройки из env:

```text
MODEL_DIR
DEFAULT_TOP_N
DEFAULT_RECENT_K
DEFAULT_MIN_COUNT
DEFAULT_CANDIDATE_POOL
CORS_ORIGINS
PUBLIC_BASE_URL
GITHUB_URL
ENABLE_FAISS
REQUIRE_MODEL_ON_STARTUP
```

`ENABLE_FAISS=1` включает построение FAISS index.

### 7.3. `schemas.py`

Описывает Pydantic-схемы запросов и ответов:

- входной текстовый трек;
- запрос рекомендаций;
- ответ рекомендаций;
- результат поиска;
- результат matching.

Лимиты для плейлиста подняты до 5000 треков, чтобы большие пользовательские файлы не ломались сразу на уровне Pydantic.

### 7.4. `model_store.py`

Хранит загруженную модель и индексы:

- `vectors`;
- `vocab`;
- `uri_to_idx`;
- `by_artist_title`;
- `by_title`;
- `counts`;
- `popularity_scores`;
- search index;
- artist index;
- FAISS index.

Также реализует:

- поиск треков;
- получение треков по URI;
- nearest-neighbor поиск через FAISS.

### 7.5. `playlist_matcher.py`

Сопоставляет текстовые строки из плейлиста с треками из MPD.

Логика:

- точное совпадение artist + title;
- fallback по title;
- статус уверенности;
- детальный ответ для frontend.

### 7.6. `recommender.py`

Главный файл алгоритма рекомендаций.

Делает:

- построение recent/whole vectors;
- получение candidate pool через FAISS или numpy fallback;
- фильтрацию seed и дубликатов;
- добавление favorite artist candidates;
- расчет score-компонентов;
- финальное ранжирование;
- генерацию объяснения рекомендации.

## 9. API

### 9.1. Health and metadata

```text
GET /health
GET /model
```

`/health` нужен для быстрой проверки, что backend жив и модель загружена.

`/model` показывает метаданные модели и индексов, включая:

```text
search_index_loaded
search_token_count
artist_index_artist_count
faiss_enabled
faiss_available
faiss_index_loaded
faiss_error
```

### 9.2. Tracks

```text
GET /tracks/search?q=...&limit=...&offset=...
POST /tracks/batch
GET /tracks/{track_uri}
```

`/tracks/search` используется frontend-поиском.

`/tracks/batch` нужен, чтобы получить метаданные пачки URI.

### 9.3. Recommendations

```text
POST /recommend/match
POST /recommend/playlist
POST /recommend/uris
```

`/recommend/match`:

- принимает artist/title;
- возвращает подробный matching;
- используется после загрузки файла.

`/recommend/playlist`:

- принимает текстовые artist/title;
- сам делает matching;
- затем строит рекомендации.

`/recommend/uris`:

- принимает уже найденные `track_uri`;
- используется frontend после того, как matching уже выполнен;
- это более контролируемый путь, потому frontend точно знает, какие треки найдены.

## 10. Docker и HTTPS

Добавлены:

```text
Dockerfile
docker-compose.yml
Caddyfile
.dockerignore
.env.example
DEPLOYMENT.md
```

Docker Compose поднимает:

1. `app` - FastAPI backend и static frontend.
2. `caddy` - reverse proxy и HTTPS.

На сервере с нормальным доменом Caddy сможет автоматически получить Let's Encrypt сертификат.

На VDI-сервере MIPT ситуация отличается:

- Docker может быть недоступен;
- публичный доступ может идти через проброшенный порт;
- HTTPS может быть самоподписанным, если нет полноценного домена и Let's Encrypt.

Поэтому для VDI был предусмотрен запуск напрямую через venv/uvicorn и скрипт:

```text
scripts/start_vdi_server.sh
```

На сервере поиск работал слишком долго.

Исправление:

Добавлены:

- нормализованные artist/title в памяти;
- token index;
- prefix lookup;
- result cache.

Результат локально:

```text
nirvana: около 0.001s
love: около 0.024s
smells like teen spirit: около 0.017s
```

### 14.7. Ускорение рекомендаций через FAISS

Проблема:

На сервере рекомендации могут тормозить из-за полного прохода по векторам.

Исправление:

Добавлен FAISS `IndexFlatIP`.

Результат локально на полной модели:

```text
Balance:  около 0.098s
Favorite: около 0.179s
```

Первый старт backend стал дольше, потому что строятся индексы, но ответы после старта стали быстрее.

## 18. Основные ограничения текущей версии

1. MPD старый, поэтому часть современных артистов и треков отсутствует.
2. Модель знает только треки из своего `vocab.jsonl`.
3. Нет live-интеграции со Spotify API.
4. Нет анализа аудио, lyrics, жанров или настроения.
5. FAISS index строится в памяти при старте, поэтому первый старт дольше.
6. Для полноценной индексации поисковиками нужен нормальный домен и валидный HTTPS.
7. Если сервер слабый, нужно следить за RAM и временем старта.
