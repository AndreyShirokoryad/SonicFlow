SonicFlow: рекомендации треков для продолжения плейлиста

SonicFlow помогает подобрать треки, которые логично продолжат уже собранный плейлист. Пользователь добавляет треки вручную или загружает файл, сервис сопоставляет их со словарем модели MPD Item2Vec и выдает top-N рекомендаций с объяснимыми компонентами score.

Что умеет сайт

  • Искать треки по названию или артисту в словаре обученной модели.
  • Показывать больше результатов поиска без перезагрузки страницы.
  • Добавлять найденные треки в верх плейлиста. Верх списка считается самым свежим концом плейлиста.
  • Загружать плейлист из TXT/CSV и быстро сопоставлять строки с треками из MPD.
  • Помечать строки загрузки как найденные, не найденные или распознанные неуверенно.
  • Строить рекомендации только по распознанным трекам, чтобы отсутствующие в базе строки не ломали расчет.
  • Настраивать режим рекомендаций, размер выдачи, recent window и минимальный MPD count для кандидатов.
  • Открывать найденные и рекомендованные треки в Spotify по URI.

Параметр Min count влияет только на кандидатов в рекомендациях. Он не ограничивает обычный поиск по названию или артисту, поэтому редкие треки все равно можно найти и добавить в плейлист.

Как устроен пользовательский сценарий

  1. Пользователь вводит название трека/артиста или загружает файл с плейлистом.
  2. Frontend отправляет строки на POST /recommend/match и получает подробный статус матчинга.
  3. В список попадают все строки, но для рекомендаций используются только треки, найденные в словаре модели.
  4. Самые свежие треки находятся сверху списка. Именно верхние распознанные треки образуют recent window.
  5. Backend считает рекомендации по выбранному режиму и возвращает score, компоненты score и короткое объяснение.

Используемая модель

Основная модель обучена на Spotify Million Playlist Dataset. В Item2Vec каждый плейлист рассматривается как последовательность токенов, а каждый трек Spotify URI - как отдельное слово. Если два трека часто встречаются в похожих плейлистах и окружениях, их embedding-векторы становятся близкими.

Модель не слушает аудио и не анализирует жанр напрямую. Главный обучающий сигнал - совместная встречаемость треков в реальных плейлистах. Поэтому рекомендации отражают не акустическую похожесть в чистом виде, а то, как люди обычно группируют треки в плейлистах.

1 188 873 треков в словаре полной локальной модели
64 размерность embedding-вектора каждого трека
1 000 000 плейлистов MPD использованы при обучении

Параметры обучения

  • Backend модели: gensim.Word2Vec.
  • Режим Word2Vec: Skip-gram, sg=1.
  • Окно контекста: window=5.
  • Negative sampling: negative=5.
  • Минимальная частота попадания в словарь модели: min_count=2.
  • Количество эпох обучения: epochs=5.
  • Файлы для backend: item_vectors_normalized.npy, vocab.jsonl, model_meta.json.

Как считается рекомендация

После матчинга backend берет URI распознанных seed-треков, находит их векторы и строит несколько признаков для каждого кандидата. Треки, уже добавленные в плейлист, из выдачи исключаются. Также отсекаются почти дубли seed-треков, например live/remaster/remix-версии с тем же артистом и названием. 

score =
  recent_weight * recent_similarity
+ whole_weight * whole_playlist_similarity
+ multi_seed_weight * multi_seed_support
+ popularity_weight * popularity_prior
+ favorite_artist_weight * favorite_artist_affinity
  • recent_similarity - похожесть на верхние Recent K распознанных треков.
  • whole_playlist_similarity - похожесть на средний вектор всего распознанного плейлиста.
  • multi_seed_support - насколько кандидат поддерживается несколькими seed-треками, а не только одним.
  • popularity_prior - нормализованная популярность трека в MPD через log1p(count).
  • favorite_artist_affinity - близость к любимым артистам в текущем seed-плейлисте.

Режимы работы

Режим меняет только веса компонентов ранжирования. Поиск треков, матчинг загруженного файла и состав словаря модели от режима не зависят.

Режим Когда использовать Веса score
Balance Режим по умолчанию. Подходит, когда нужно продолжить плейлист без сильного перекоса в свежесть, популярность или любимых артистов. Recent 45%, whole playlist 25%, multi-seed 20%, popularity 10%, favorite artist 0%.
More recent Лучше подходит, если верхние треки списка задают текущее настроение и именно их нужно продолжить. Особенно полезен после ручного добавления нескольких новых треков через поиск. Recent 65%, whole playlist 15%, multi-seed 15%, popularity 5%, favorite artist 0%.
Favorite Сильнее учитывает артистов, которые повторяются в плейлисте. Режим поднимает треки этих исполнителей и похожих по artist-affinity кандидатов, даже если общий MPD count у них небольшой. Recent 25%, whole playlist 20%, multi-seed 15%, popularity 5%, favorite artist 35%.
More popular Делает выдачу более узнаваемой и устойчивой за счет треков, которые часто встречались в MPD. Подходит, когда нужны менее рискованные рекомендации. Recent 35%, whole playlist 20%, multi-seed 20%, popularity 25%, favorite artist 0%.

FAISS-ускорение

Для полной модели перебор всех векторов на каждый запрос был бы тяжелым. Поэтому backend при старте строит FAISS IndexFlatIP по нормализованным векторам. Так как векторы нормализованы, inner product эквивалентен cosine similarity. FAISS быстро находит ближайших кандидатов для recent-вектора и whole-playlist вектора, а итоговое ранжирование уже считается только по этому candidate pool.

Если faiss-cpu недоступен или ENABLE_FAISS=0, сервис остается рабочим и переходит на chunked NumPy-перебор. Это медленнее, но сохраняет корректность результата.

Поиск и матчинг

Поиск по сайту использует отдельный token/prefix index по нормализованным названиям треков и артистов. Запросы кешируются, поэтому повторный поиск популярных строк работает быстрее. Для загрузки файла используется batch matching: frontend отправляет строки пачками, а backend пытается найти точное совпадение artist/title, затем более мягкие совпадения по названию и артисту.

Неуверенно распознанные строки отображаются отдельно, чтобы пользователь мог проверить их вручную. В рекомендации они могут попасть только если backend вернул конкретный track_uri.

Основные API endpoints

  • GET /health - статус приложения и факт загрузки модели.
  • GET /model - метаданные модели, search index, artist index и состояние FAISS.
  • GET /tracks/search?q=... - поиск треков по названию или артисту.
  • POST /tracks/batch - получение данных по списку Spotify URI.
  • POST /recommend/match - подробное сопоставление строк плейлиста с MPD-моделью.
  • POST /recommend/playlist - рекомендации по текстовому плейлисту.
  • POST /recommend/uris - рекомендации по уже известным Spotify URI.

Запуск и публикация

Проект подготовлен к локальному запуску через Python/FastAPI и к production-запуску через Docker Compose. FastAPI отдает API, frontend, /documentation, /robots.txt и /sitemap.xml. Caddy используется как внешний reverse proxy и может автоматически выпускать HTTPS-сертификат для публичного домена.
